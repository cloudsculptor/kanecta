Architecture & Strategy

From Knowledge Graph to Organisational Intelligence

How Kanecta becomes the connective layer that makes your organisation's knowledge queryable, persistent, and AI-ready.

The Problem

Knowledge accumulates in silos

Most organisations running large, long-lived software projects scatter their knowledge across disconnected tools — tickets in Jira, documentation in Confluence, code in GitHub, decisions in people's heads. These systems don't talk to each other in any meaningful way.

An engineer joining the team can read the tickets and the docs, but has no way to understand why things are the way they are, how decisions connect to code, or what was tried and rejected.

This is a solvable problem. The solution is a knowledge graph — a unified, structured store where everything connects to everything, and where the reasoning behind decisions is captured alongside the decisions themselves. Kanecta is that knowledge graph.
Differentiation

Everything is an item, and everything connects

Most tools optimise for one dimension — Jira for task tracking, Confluence for documentation, Notion for notes, Neo4j for graph queries. Each is best-in-class at its slice and blind to everything else.

In Kanecta, a Jira ticket is an item. An architectural decision is an item. A code component is an item. A team member's area of expertise is an item. The relationship between a ticket and the architectural decision that constrains it is also an item — typed, directional, carrying its own metadata and reasoning.

🔗

Cross-silo queries

"What architectural decisions are upstream of this component?" is a graph traversal. Unanswerable in siloed tools. Trivial in Kanecta.

🧠

Reasoning captured

Decision items record what was decided, why, what was rejected, and who decided. Six years of that compounds into irreplaceable institutional memory.

🤖

AI reads and writes

The same store humans maintain is the store AI agents draw from — and contribute back to. Every agent insight is persisted for next time.

Technical Foundation

Three capabilities, one database

Kanecta runs on Postgres — giving it three complementary search and query capabilities without managing multiple systems. The combined pipeline means the right information surfaces even when connections are implicit, terminology varies, or the explicit relationship was never created.

Relational · SQL

Structured queries

Standard SQL for filtering, joining, and aggregating. Items, relationships, history, types — all queryable with full SQL expressiveness.

Graph · Apache AGE

Graph traversal

Cypher-based multi-hop queries. "Find everything connected to this ticket within three hops." Single statement. Any depth.

Vector · pg_vector

Semantic search

Vector embeddings on every item. "Find items whose meaning is close to this concept" — regardless of whether the exact words match.

AI Agents

The difference between assisting and compounding

Today, AI agents working on software tasks start cold. They read a ticket, they read some code, they make their best guess at what matters. They have no memory of what was tried before, no access to the reasoning behind past decisions.

With a rich Kanecta knowledge graph, an agent working on a ticket can:

  1. Traverse the graph from the ticket UUID — reaching acceptance criteria, related design decisions, affected components, component owners, past PRs, known constraints.

  2. Surface semantic matches — past decisions and patterns that share meaning but weren't explicitly linked, found via vector similarity.

  3. Arrive with full context — not just what the ticket asks for, but why this approach, what was rejected, who to consult, what's fragile nearby.

  4. Write back — recording decisions made, constraints discovered, relationships created — so the next agent or human finds the graph richer than they left it.

Adoption Pathway

Four stages to organisational intelligence

1

Core team — cloud knowledge graph

A small core team begins using Kanecta with a shared Postgres instance on AWS, configured with Apache AGE and pg_vector. The foundation: a live, queryable knowledge graph replacing informal knowledge held in people's heads.

  • Architectural decisions with full reasoning
  • Relationships between tickets, Confluence pages, and code components
  • Team expertise mapping — who knows what, who owns what
2

Wider team access

Architects, business analysts, product owners, and tech leads are added. Each brings a different angle of knowledge that enriches the graph. The value begins to compound.

  • BA captures business reasoning — AI agents can traverse it later
  • Architects record technology decisions — context for every engineer
  • Non-technical users contribute through the UI — no technical knowledge required
3

Full organisation

All engineers and non-engineers onboarded. Jira, Confluence, and GitHub are modelled as items in the graph — tickets, pages, and pull requests become first-class nodes connected to the knowledge that explains them.

  • Kanecta becomes the connective tissue of organisational knowledge
  • Not replacing Jira or Confluence — making them navigable together
4

Agentic pipelines

With a rich knowledge graph in place, agentic pipelines become viable. A Jira-to-PR pipeline — where an AI agent receives a ticket, traverses the graph, implements the change, and writes its reasoning back — is an engineering task with clear infrastructure.

Jira ticket moves to Ready
→ Agent traverses knowledge graph (ticket → decisions → components → constraints)
→ Agent plans, writes decision item to graph
→ Agent implements, creates PR
→ PR linked to ticket in graph
→ Decisions and discoveries written back

Each run makes the graph richer. Each subsequent agent — or human — working in the same area finds better context than the last.

Timing

Why now

🧩 AI capability

Large language models can now traverse structured context and produce code that reflects genuine understanding. The bottleneck is no longer capability — it is context. A knowledge graph is context infrastructure.

🔧 Tooling maturity

Postgres, Apache AGE, pg_vector, the Model Context Protocol, LangGraph — all production-ready. This is not prototype technology.

⏳ Cost of waiting

A large, long-lived project accumulates knowledge debt in proportion to its size and duration. The longer it runs without a structured knowledge layer, the more expensive every transition, onboarding, and change becomes.

Six years of a well-maintained Kanecta knowledge graph is a compounding asset. Every decision captured is context available to every future engineer, analyst, and AI agent who works on the project.

The graph does not make the organisation dependent on any vendor. It is self-hosted, open-source, owned entirely by the organisation. The knowledge is yours.

The question is not whether a knowledge graph is valuable.
The question is whether to build it now or pay a larger price later.
