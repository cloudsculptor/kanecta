= Sub-projects

== kanecta-api

A Node.js/Express HTTP server that exposes datastore contents over a REST API.

*Tech:* Node.js, Express 4, Jest, Supertest

=== Endpoints

[cols="1,2,3"]
|===
| Method | Path | Description

| `GET` | `/items/:id` | Returns the metadata for a single item
| `GET` | `/items/:id?levels=N` | Returns the item and its descendants as a nested tree, up to N levels deep
|===

`levels=1` returns the item alone, `levels=2` includes direct children, `levels=3` includes grandchildren, and so on.

=== Running

[source,bash]
----
cd kanecta-api
npm install
npm start          # listens on port 3000
----

=== Environment variables

[cols="2,2,3"]
|===
| Variable | Default | Description

| `PORT` | `3000` | Port the server listens on
| `KANECTA_CONFIG` | platform default (`~/.config/kanecta/config.json`) | Path to the config that selects the datastore / working set
|===

=== Tests

[source,bash]
----
npm test
----

== kanecta-cli

A Node.js script that reads the datastore and writes a hierarchical plain-text file.

*Tech:* Node.js (no dependencies)

=== Usage

[source,bash]
----
cd kanecta-cli
npm run kanecta
----

Reads the `defaultView` from the datastore's `config.json` (root item ID and max depth), walks the tree, and writes `kanecta.txt` to the datastore root.

== kanecta-client-web

A React/TypeScript single-page app for browsing a Kanecta datastore through the API.

*Tech:* React 19, TypeScript 6, Vite 8, Sass, ESLint, Prettier, Storybook 10

=== Running in development

Start the API server first (the web client proxies all `/api` requests to `localhost:3000`):

[source,bash]
----
# terminal 1
cd kanecta-api && npm start

# terminal 2
cd kanecta-client-web
npm install
npm run dev        # http://localhost:5173
----

=== Other scripts

[source,bash]
----
npm run build          # type-check + production build
npm run lint           # ESLint
npm run format         # Prettier (writes)
npm run format:check   # Prettier (check only)
npm run storybook      # component explorer on http://localhost:6006
npm run build-storybook
----

=== How it works

The app fetches a configurable root item from the API and renders it as an interactive tree. A number input controls how many levels of descendants are loaded — changing it immediately triggers a new fetch.

The Vite dev server proxies `/api/*` to `http://localhost:3000`, so no CORS setup is needed during development.

== Test fixtures (generated, not committed)

There is no committed sample datastore. A committed fixture freezes one on-disk
format and rots whenever the format changes, so test datastores are *generated*
from the current adapter instead — they always match the live format.

* *Per suite, on the fly.* Most suites seed a fresh datastore in a temp directory
  in their setup (e.g. a `tmpDs()` helper calling `Datastore.init` + `create`).
* *Shared helper where reused.* Suites that need the same populated read-only
  store use `kanecta-storage-adapters/kanecta-sqlite-fs/test-helpers/makeSampleDatastore.js`,
  which builds a known "Base Work Process" tree and returns the generated UUIDs
  (reference them via the returned `ids`, never by literal). The CLI tests use it.

The fixtures track the spec automatically because they run `init`/`create`, which
implement it — the spec stays the contract, not a stored datastore.
