The storage levels (L0 / L1 / L2)

Every adapter has the same three roles, numbered from the source of truth (0) outward to the most derived and disposable (2). The number tracks the role, not the medium — so L1 means the same thing everywhere. Lower number = closer to the truth; everything above L0 is derived and 100% rebuildable.

Filesystem + SQLite

L0 · Authority — source of truth
item.json + file sidecars
L1 · Structured store
SQLite — complete copy of every item.json (derived)
L2 · Projections
CQRS read tables (derived)

Postgres + S3

L0 = L1 · Authority & Structured store
item tables + S3 objects (the DB is the structured store)
L2 · Projections
CQRS read tables (derived)

Adapters may collapse adjacent levels: Postgres has no separate derived copy — L0 and L1 are the same tables.

L0 Authority — must be atomic L1 Structured store — derived L2 Projections — derived

The life of a write

Writes to a branch are serialized (one in flight at a time; new writes queue). A write either fully lands or fully rolls back, and is reported successful only after it lands.

  1. queue — wait for the branch's write slot (serialized; new writes queue, none dropped)
  2. lock — take the cross-process lock (PID + heartbeat)
  3. journal — record intent + the adapter's full step plan (write-ahead)
  4. L0 — write authority (item.json + sidecars / S3 then the Postgres txn)  — the one step that must be atomic
  5. L1 — update the structured store (SQLite copy; = L0 on Postgres)
  6. L2 — update projections
  7. commit — mark the journal committed, release the lock
  8. return — only now is success reported to the caller

Crash before step 7? On next open: if L0 completed → roll forward (rebuild L1/L2 from L0); if L0 did not complete → roll back to the pre-image. Because L1/L2 are 100% derived, only the L0 step must be truly atomic.

Reads during a write? Never blocked, never partial — a snapshot read returns the last fully-committed value (at most one version stale, only during the write window). Read-your-writes holds because writes are synchronous. The new version becomes visible atomically across the levels a reader touches.

Why a journal at all? No single transaction spans filesystem + SQLite, or Postgres + S3 — the journal is what makes a multi-store write atomic and recoverable.

Branches: per-branch folders & classes

Every branch — including main — is a complete, self-contained folder. Kanecta branching is independent of Git.

.kanecta/
  config/config.json                 datastore-level config (shared)
  branches/<encoded-name>/
    items/<shard>/item.json           the branch's items (added / edited / deleted_at tombstone)
    index.db                         per-branch derived index (never a source of truth)
    branch.json                      { name, fill, upstream, branchPoint, createdAt }

A read is a layer stack: branch diff → working-set data → device library (core/app, read-only). Membership, point reads, navigation, and merge are exact; cross-seam sort/paging/search-ranking is best-effort.
