Comparisons with Other Tools

Kanecta vs Neo4j.

Neo4j is the most mature property graph database. Kanecta's vNext spec describes its relationship model as nearly mechanical to export to Neo4j. Here is an honest assessment of the fit — and which model serves AI agents better.

Neo4j as a datastore for Kanecta

Kanecta's vNext spec mentions Neo4j by name. The first-class relationships design describes the export as "almost mechanical" — relationship items have source, target, and a type label, which maps directly to Neo4j edges:

"Export is mechanical — one Cypher statement per relationship item: MATCH (a {id: $source}), (b {id: $target}) CREATE (a)-[:DEPENDS_ON { label: $label }]->(b)"

Where Neo4j is genuinely strong

Cypher is more readable and expressive than SQL for graph patterns. Multi-hop traversal — "find everything reachable from X within N hops matching label Y" — is one query. Neo4j also ships built-in full-text search via Lucene indexes, which means no separate OpenSearch cluster. And the tooling (Neo4j Browser, Bloom) is excellent for visual exploration.

Maps well
  • ✓ Multi-hop relationship traversal — one Cypher query
  • ✓ Backlink lookups — graph traversal over inline [[uuid]] references
  • ✓ ReBAC permission checks — Zanzibar-style graph walk over permission items
  • ✓ Built-in FTS via Lucene — no separate service needed
  • ✓ Expressive Cypher over Gremlin
  • ✓ The vNext spec already describes the export shape
Falls apart
  • ✗ Edges cannot have edges — the spec calls this out as the known mismatch; relationship items can themselves be related in Kanecta
  • ✗ Per-type tables (obj_<uuid>) — no equivalent; type-level constraints and indexes are lost
  • ✗ History and the activity log — not graph problems; Neo4j gives you nothing over a flat table for bitemporal queries
  • ✗ Schema migrations — no equivalent to numbered .sql files run in order
  • ✗ Not AWS-native — AuraDB adds cross-region latency; self-hosting on EC2 adds ops burden
Verdict

Neo4j covers the graph traversal cases well and is a better fit than Neptune. But Kanecta is primarily a rich entity store with graph features, not a graph database with entity features. The 80% that is entity queries, history, per-type tables, and migrations still fights the model. Neo4j belongs as a projection index — exported from Postgres, rebuildable on demand — not the primary store.

Which model is better for AI?

Kanecta is explicitly designed as a human-AI bridge. The spec states it plainly: "structured enough for AI to work with efficiently, transparent enough for humans to audit and understand." How does each model serve an AI agent?

Conclusion
Kanecta's model wins for AI.
Not because it's a graph — because it's consistent.
Kanecta (items model)
Winner
  • One mental model, no special cases Types, relationships, permissions, history, functions — all items. An AI learns the model once and applies it everywhere. No context-switching between node/edge/label concepts.
  • The system describes itself Types are items. Functions are items. An AI can read a type item to discover what fields its instances have, then act on both through identical operations. Neo4j's schema is implicit.
  • Function items Stored functions are first-class items. The vNext spec notes: "A stored function doesn't have to be executable code — it can be a description that Claude executes at runtime." AI can discover and execute stored behaviour through the same interface it uses for everything else.
  • Decision items The decision type captures what was decided, why, what was rejected, and tradeoffs. Institutional memory designed for AI reasoning, not just retrieval.
  • Aliases and value Every item has a human-readable value and optional named aliases. AI can refer to items by name rather than UUID, always has a display string without a secondary lookup.
  • MCP interface abstracts storage An AI agent never writes SQL or Cypher. It calls kanecta_get, kanecta_search, kanecta_relate. The underlying store is invisible. What matters is the conceptual model — and Kanecta's is simpler.
Neo4j (property graph)
One real advantage
  • Complex multi-hop traversal "Find everything reachable from X within 3 hops matching label Y" is one Cypher query. In Kanecta it requires a recursive CTE or multiple MCP calls. If the primary AI use case is deep knowledge graph exploration, Neo4j wins this specific pattern.
  • Natural language maps to graph Questions like "how is A related to B?" feel natural to express in Cypher. An AI writing graph queries directly would find Cypher readable.
  • Explicit node/edge distinction In theory, typed edges vs nodes is cleaner for AI reasoning. In practice, Kanecta's uniform item model is simpler to hold in a context window.
  • Visual exploration Neo4j Bloom is good for human-AI collaborative exploration. Less relevant for automated AI agent use.

Graph traversal is one query type. Kanecta's AI workload is the full lifecycle of knowledge: capture, structure, relate, reason about, execute, audit. The items model serves all of it through a single consistent interface. Neo4j serves one of those operations better and fights the rest.

The strongest framing comes from the vNext spec itself. Kanecta independently arrived at the same place as Smalltalk ("one kind of thing, consistent rules, no special cases"), RDF (first-class relationships, reification), and property graphs (typed edges with properties) — without the baggage of any of them. The payoff: a simpler storage model, a simpler query model, and a system that gets more powerful as it grows rather than more complicated.
