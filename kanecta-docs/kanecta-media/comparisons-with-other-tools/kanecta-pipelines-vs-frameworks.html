Introduction

Data-first vs code-first

Every external orchestration framework — LangGraph, LangChain, CrewAI, Mastra, Inngest — defines pipelines as code that lives in a repository. A pipeline is a Python class or a TypeScript function. It runs, it evaporates, and its history lives in a separate observability product if at all.

Kanecta pipelines are data. A pipeline item lives in the same datastore as the findings it audits, the decisions it informs, and the code it implements. Its run history accumulates as pipeline-run items you can query with the same tools you use for everything else. An agent can read a pipeline definition and reason about it. A non-technical user can open it in Studio and edit it without touching a codebase.

This is a structural difference, not a feature difference. The comparisons below trace how that structural choice plays out against fourteen of the most widely used frameworks.

Where Kanecta wins structurally

The advantages no other framework replicates

🗃

Knowledge graph persistence

Every pipeline, run, finding, decision, and agent is a queryable item in the same datastore. "Show all runs of pipelines relating to the auth decision" is a trivial query. In any other framework this relationship does not exist as a concept.

🚦

Three-family confidence gates

accuracy: gates are fully automated (AI self-assessment). review: gates require per-item human sign-off (lifecycle state). manual gates are unconditional human stops. No other framework models human review at this granularity as a first-class workflow primitive.

🧪

Six-type eval system

count, type, payload, meta, llm-judge, and no-regression assertions all tracked as first-class items in the same tree as runs. Mastra has evals; LlamaIndex has evals. Neither tracks them alongside the run history they test.

🔄

Full cycle model

loopBack, maxIterations, and the global iteration counter give Kanecta LangGraph-comparable cycle support with the audit trail that LangGraph's in-memory state cannot provide — every iteration's items are preserved and tagged.

🤖

Group-chat runtime

Multi-agent deliberation (a panel of agents with different models and roles) is a first-class runtime — not a bespoke pattern you build yourself. From the pipeline's perspective it is identical to any other phase: it has gates, loops, fan-out, and heartbeats.

🔓

Provider independence

runtime is an open string. Claude, GPT-4o, Gemini, a local Ollama instance, a Python script, a kanecta-function, an HTTP call — all are first-class without code changes to the pipeline definition itself.

Group A — AI orchestration frameworks

LangChain, LangGraph, CrewAI, AutoGen, Mastra, LlamaIndex, Haystack, OpenAI Agents SDK

These frameworks define pipelines as code — Python or TypeScript. They have no datastore; a pipeline run evaporates when the process ends unless you bolt on external observability.

LangChain

AI orchestration · MIT
Kanecta wins clearly
Where LangChain is strong
  • Integration library — 200+ loaders, tools, vector store integrations. Unmatched breadth.
  • Community — largest AI framework community; most StackOverflow answers, most tutorials.
  • Prototyping speed — chain a prompt to a retriever in five lines of Python.
Where Kanecta wins
  • Run history is queryable data, not ephemeral stdout.
  • Human gates, confidence levels, and human review lifecycle are native primitives.
  • No external observability product needed — Studio shows run state from the same datastore.
  • Non-technical users can read and edit pipelines without a Python environment.
Kanecta adds: Persistent run history Human gates Evals as items Studio editing Triggers

LangGraph

Graph-based AI orchestration · MIT
Closest technical match
Where LangGraph is strong
  • Typed state dict — every node receives typed state and must return typed state. Compile-time correctness that Kanecta's untyped phase output cannot match yet.
  • Subgraphs — reusable graphs embedded inside larger graphs; production-ready and well-tested.
  • Native cycles — graph cycle model is first-class; Kanecta's loopBack was modelled after it.
  • Streaming — token-level streaming from nodes to the caller.
Where Kanecta wins
  • Persistent, queryable run history. LangGraph's optional checkpointing requires you to wire in a checkpoint store yourself.
  • Three-family confidence gates — LangGraph's interrupt is a single binary pause with no item-level confidence model.
  • Phase output is persisted on the run record and inspectable after the fact; LangGraph state is in-memory.
  • Evals as first-class items alongside runs — no LangSmith subscription needed.
  • Reactive triggers and concurrency policy — no equivalent in LangGraph.
  • Provider-independent — LangGraph is built for LangChain's model abstractions.
LangGraph still ahead on: Typed state Streaming Kanecta adds: Persisted phase output 3-family gates Evals Triggers

CrewAI

Role-based multi-agent · MIT
Kanecta wins
Where CrewAI is strong
  • Role/goal model — agents have explicit roles, goals, and backstories. Intuitive for teams thinking in terms of "who does what".
  • Task delegation — agents can delegate tasks to other crew members mid-execution.
  • Crews as a concept — grouping agents into named crews is a good mental model for non-engineers.
Where Kanecta wins
  • Kanecta's runtime: "group-chat" provides the multi-agent deliberation pattern more flexibly — any set of agents, any model mix, any termination condition.
  • CrewAI has no data persistence; run output evaporates.
  • No human gates, no confidence system, no evals.
  • CrewAI Studio is a paid product; Kanecta Studio is open source.
Kanecta adds: group-chat runtime Persistence Gates & confidence Evals Free Studio

AutoGen (Microsoft)

Conversational multi-agent · MIT
Kanecta wins
Where AutoGen is strong
  • Conversable agents — any agent can initiate a conversation with any other agent. Flexible for ad-hoc delegation patterns.
  • Human-in-the-loop — a UserProxyAgent can interject at any conversation turn.
  • Code execution — agents can write and execute code in a sandboxed environment within the conversation.
Where Kanecta wins
  • Kanecta's group-chat runtime covers AutoGen's conversational pattern with a structured termination model (explicit vs max-turns).
  • AutoGen has no data persistence, no confidence system, no evals, no triggers.
  • AutoGen runs evaporate; Kanecta's transcript is stored on the run record as a queryable sidecar.
  • AutoGen is OpenAI-optimised; Kanecta is provider-independent.
Kanecta adds: Structured termination Inspectable transcript Persistence Gates Provider independence

Mastra

TypeScript-first AI orchestration · MIT
Closest TS competitor
Where Mastra is strong
  • TypeScript-native — typed workflows with full TS inference. No Python dependency.
  • Built-in evals — first-class evaluation framework; one of very few orchestrators with this.
  • Nested workflows — workflows can call other workflows as steps.
  • Agents & tools — clean agent/tool/workflow separation similar in spirit to Kanecta.
Where Kanecta wins
  • Knowledge graph persistence — Mastra manages its own state but it is not a queryable knowledge graph alongside domain data.
  • Three-family human confidence gates — Mastra has no equivalent of the review lifecycle.
  • Kanecta evals are tracked as items in the same tree as runs; Mastra evals are a framework feature, not persistent records.
  • Reactive triggers and concurrency policy are partial in Mastra.
Mastra ahead on: TypeScript inference Kanecta adds: Knowledge graph Human review lifecycle Evals as items

LlamaIndex

RAG / retrieval framework · MIT
vNext adapter planned
Where LlamaIndex is strong
  • Sub-question decomposition — breaks a complex question into targeted sub-questions, answers each, synthesises results.
  • Recursive retrieval — queries a summary index first, drills into relevant chunks.
  • 100+ data loaders — PDF, Notion, Confluence, databases, web pages, GitHub.
  • RouterQueryEngine — routes different query types to different strategies automatically.
Why they compose rather than compete
  • LlamaIndex is a retrieval library, not an orchestration framework. It does not replace pipeline scheduling, gates, or evals.
  • Kanecta agents have full MCP tool access and can make multiple complex queries during execution — for data already in Kanecta, sub-question decomposition is achievable natively.
  • The gap is for external document collections not yet in Kanecta — PDFs, Confluence spaces, databases. The vNext LlamaIndex adapter bridges this: ingest external docs as Kanecta items via LlamaIndex loaders, then query them with LlamaIndex strategies inside Kanecta pipeline phases.
See vNext: kanecta-llamaindex-ingest runtime: "llamaindex"

Haystack (deepset)

NLP / retrieval pipeline · Apache 2.0
vNext adapter planned
Where Haystack is strong
  • Hybrid dense + sparse retrieval — BM25 + vector, reciprocal rank fusion. Best-in-class for keyword-sensitive domains (legal, compliance, technical docs).
  • Document preprocessing — mature, battle-tested pipeline for cleaning, chunking, deduplication.
  • On-premise / regulated industries — designed to run entirely self-hosted with open-source embedding models.
Why they compose
  • Same reasoning as LlamaIndex — Haystack is a retrieval library, not an orchestration framework.
  • The vNext adapter adds runtime: "haystack" to agentPayload, using Haystack for retrieval and Kanecta for orchestration, gates, and persistence.
  • Combined with Kanecta running on-premise, this gives a fully self-hosted AI pipeline stack — the strongest positioning for regulated industries.
See vNext: kanecta-haystack-ingest runtime: "haystack" hybrid search (search.sparse)

OpenAI Agents SDK

Handoff-based agent orchestration · MIT
Kanecta wins significantly
Where the SDK is strong
  • Handoffs — the most elegant delegation primitive of any framework; one agent passes control to a specialist with full context in a single call.
  • Parallel guardrails — a fast cheap validation model runs concurrently with the main agent and can trip a wire before the expensive model finishes.
  • First-party tracing — zero-config observability if you are already on the OpenAI dashboard.
Where Kanecta wins
  • No persistence — run output lives only in the OpenAI dashboard.
  • No cycles, no fan-out, no fan-in, no gates, no evals, no triggers.
  • OpenAI-first; other model providers require adapters that are not first-class.
  • Kanecta's fallbackAgentId covers handoffs; group-chat covers multi-agent deliberation; a pre-phase with confidenceGate covers guardrails (with the added benefit that failures are tracked items, not lost events).
See vNext: runtime: "openai-agents" adapter Kanecta adds: Persistence Cycles Fan-out Gates Evals
Group B — Pipeline & workflow engines

Dify, Temporal, n8n, Windmill, AWS Step Functions, Inngest

These are the closest structural competitors to Kanecta — all provide durable or persistent workflow execution, visual editors, or production-grade scheduling. Kanecta competes with this group on orchestration while adding AI-specific features they lack entirely.

Dify

Visual AI workflow builder · Apache 2.0
Most direct competitor
Where Dify is strong
  • Visual pipeline editor — the best drag-and-drop AI workflow builder available; genuinely accessible to non-engineers.
  • Knowledge base integration — first-class RAG built into the workflow, not an external add-on.
  • Sub-workflows — reuse a workflow as a node inside another workflow.
  • Triggers — webhook triggers, API triggers, chat mode.
  • Large community — 40,000+ stars, active ecosystem.
Where Kanecta wins
  • Knowledge graph persistence — Dify manages its own state; Kanecta outputs are first-class queryable items alongside all your other data.
  • Three-family confidence gates — Dify has no human review lifecycle or per-item confidence system.
  • Six-type evals including llm-judge — Dify's eval system is basic.
  • Provider-independent runtime — Dify has model preferences baked in.
  • Group-chat runtime — Dify has no equivalent of structured multi-agent deliberation.
  • Cycles with full iteration tracking — Dify's loop support is partial.
Dify ahead on: Visual editor Knowledge base UX Community size Kanecta adds: Knowledge graph Confidence gates Evals group-chat

Temporal

Durable execution engine · MIT server
Stronger execution guarantees today
Where Temporal is strong
  • Durable execution — the gold standard. Workflows survive process crashes, network partitions, and server restarts with zero data loss. Battle-tested at massive scale.
  • Activity heartbeats — long-running activities heartbeat to prove they are alive; Temporal retries automatically on missed heartbeats.
  • Versioning — explicit workflow versioning so in-flight runs survive definition changes without breaking.
  • Saga pattern — compensating transactions for rollback on failure.
  • Schedules and signals — rich trigger model including cron, signal-based wake-up, and time-travel debugging.
Where Kanecta wins
  • AI-specific features — confidence gates, evals, group-chat, llm-judge, escalation. Temporal has none of these.
  • Knowledge graph persistence — Temporal tracks workflow execution; Kanecta tracks workflow execution and everything the workflow operates on.
  • Non-technical editing — Temporal workflows are code; Kanecta pipelines are Studio-editable items.
  • Provider-independent model selection — Temporal is model-agnostic because it does not invoke models at all; Kanecta is AI-native and model-agnostic.
Temporal still ahead on: Durable execution at scale Activity-level heartbeats (Kanecta has phase-level) Explicit versioning Kanecta adds: AI gates & evals Knowledge graph Studio editing

n8n

Visual workflow automation · Source-available
Different audience, some overlap
Where n8n is strong
  • Integration library — 400+ pre-built integrations (Slack, GitHub, Jira, Salesforce, …). Kanecta's runtime: "http" covers the general case but n8n's typed connectors are much faster to configure.
  • Visual editor — polished drag-and-drop with inline data preview.
  • Sub-workflows — workflows can call other workflows as nodes.
  • Trigger library — webhook, schedule, email, file system, database changes, and more.
Where Kanecta wins
  • AI-specific primitives — confidence gates, group-chat, evals, llm-judge. n8n has basic AI nodes but no confidence system or structured multi-agent patterns.
  • Knowledge graph persistence — n8n stores execution logs, not queryable knowledge items.
  • Human review lifecycle — n8n has a wait node for pausing; Kanecta's confidence gate system is far more granular.
n8n ahead on: Integration library (400+) Visual editing maturity Kanecta adds: AI gates Knowledge graph Evals group-chat

Windmill

Script & workflow runner · AGPLv3
Different primary audience
Where Windmill is strong
  • Script execution — run Python, TypeScript, Go, Bash scripts with a polished visual editor and flow builder.
  • Approval steps — a flow can pause and send an email/Slack message asking for approval before continuing.
  • Secrets & resource management — first-class handling of credentials and connection objects.
  • Scheduling — cron-based scheduling is polished and production-grade.
Where Kanecta wins
  • AI-specific primitives — confidence gates (three families), evals, group-chat, escalation. Windmill's approval step is a single binary pause with no confidence model.
  • Knowledge graph — Windmill stores run logs; Kanecta stores structured knowledge items.
  • Provider-independent AI execution — Windmill's AI support is a general HTTP call; Kanecta's runtimes are designed for AI workflows.
Windmill ahead on: Secrets management Polyglot scripts Kanecta adds: AI gates Knowledge graph Evals

AWS Step Functions

Cloud-native workflow engine · Proprietary
Different category — cloud-native vs local-first
Where Step Functions is strong
  • Managed durability — AWS guarantees exactly-once execution, automatic retries, and state persistence at scale without you running anything.
  • EventBridge integration — rich event-driven trigger ecosystem; hundreds of AWS service events can start a workflow.
  • Nested state machines — composable via child state machine invocations.
  • Workflow Studio — low-code visual editor for the Amazon States Language.
Where Kanecta wins
  • AI-specific primitives — confidence gates, evals, group-chat, escalation, heartbeats. Step Functions has no AI orchestration concepts.
  • Local-first — Kanecta runs on your machine or your server; Step Functions requires an AWS account and routes all workflow state through AWS infrastructure.
  • Knowledge graph — Step Functions tracks execution status; Kanecta tracks execution status and all the domain knowledge the workflow operates on.
  • Provider-independent — Step Functions connects to AWS services natively; Kanecta is cloud-agnostic.
Step Functions ahead on: Managed infrastructure AWS ecosystem integration Scale guarantees Kanecta adds: AI gates & evals Local-first Knowledge graph

Inngest

TypeScript durable functions · Source-available
Strong TypeScript DX, different model
Where Inngest is strong
  • TypeScript DX — step.run(), step.sleep(), step.waitForEvent() — the cleanest TypeScript API of any durable workflow framework.
  • step.sleep() — a step can sleep for hours or days without holding resources. Useful for "send email, wait 3 days, follow up" patterns.
  • Event-driven — workflows start from events; the event system is first-class and well-designed.
  • Fan-out — inngest.createFunction + parallel steps provides fan-out, though without Kanecta's maxConcurrent control.
Where Kanecta wins
  • AI-specific primitives — Inngest has no confidence gates, no evals, no group-chat, no escalation.
  • Knowledge graph — Inngest tracks function execution; Kanecta tracks execution and knowledge.
  • Provider-independent — Inngest is TypeScript-only; Kanecta supports Python, Claude Code, group-chat, HTTP, and any future runtime.
  • Kanecta's phases[].waitBefore pattern and triggers[].schedule cover similar scheduling patterns to Inngest's step.sleep.
Inngest ahead on: TypeScript DX step.sleep (arbitrary duration) Kanecta adds: AI gates & evals Knowledge graph Multi-runtime
Honest assessment

Where Kanecta still has gaps

Integration library
n8n and Inngest have hundreds of pre-built connectors with typed, validated configs for specific APIs. Kanecta's runtime: "http" covers the general case but requires hand-writing headers and body structures. This is an ecosystem gap, not a spec gap — runtime: "http" provides the primitive; typed agent configs for common APIs (Slack, Jira, GitHub, Linear) would make it production-grade.
Visual pipeline editing
Dify, n8n, Windmill, and AWS Step Functions have production-quality drag-and-drop pipeline builders. Kanecta Studio currently renders and reads pipelines but editing requires JSON. Studio editing is a planned implementation feature — the spec fully supports it (pipeline is a first-class item); the UI just needs to be built.
Typed state passing
LangGraph's typed state dict provides compile-time guarantees that every node receives and returns the correct shape. Kanecta's phases[].outputs documents intent but the runner does not enforce types at execution time. Runtime validation of declared phase outputs is a spec-level gap worth closing in a future version.
Durable execution at scale
Kanecta's spec now mandates resumption from the last completed phase on restart (via pipelineSnapshot and the run record as source of truth). But Temporal's durable execution model has years of battle-testing at massive scale. The runner implementation needs to live up to the spec commitment before this gap closes.
step.sleep / voluntary pause
Inngest's step.sleep(duration) lets a workflow step sleep for hours or days without holding resources — ideal for "send, wait, follow up" patterns. Kanecta's scheduled triggers and concurrencyPolicy: "queue" cover some of this but there is no direct equivalent of a phase voluntarily sleeping for an arbitrary duration without consuming a runner slot.
vNext adapters

Planned adapters that close retrieval and runtime gaps

Rather than rebuilding mature retrieval libraries from scratch, Kanecta's vNext plan is to provide adapter runtimes that wrap best-in-class external tools inside Kanecta pipeline phases — Kanecta handles orchestration, gates, persistence, and evals; the adapter handles retrieval or agent execution.

LlamaIndex adapter

runtime: "llamaindex" uses LlamaIndex query strategies (sub-question, recursive, router) for retrieval before model invocation. A matching ingest adapter (kanecta-llamaindex-ingest) brings external documents into Kanecta as items with embeddings.

Haystack adapter

runtime: "haystack" uses Haystack's hybrid dense+sparse retrieval before model invocation. Best for regulated industries running entirely on-premise. The ingest adapter handles document preprocessing and BM25 sparse encoding. A search.sparse field on the item spec is also flagged for vNext.

LangGraph adapter

runtime: "langgraph" wraps a LangGraph compiled graph inside a Kanecta pipeline phase. Teams with existing LangGraph agents can migrate incrementally: wrap the graph in a Kanecta phase, add Kanecta gates and evals around it, and eventually rewrite phases natively.

OpenAI Agents SDK adapter

runtime: "openai-agents" wraps an OpenAI Agents SDK run inside a Kanecta phase. Internal handoffs are opaque to Kanecta (the adapter captures final output); SDK guardrails can be optionally enabled. Useful for teams with existing SDK code who want Kanecta's orchestration and persistence without rewriting agent logic.
