Architecture Overview

Kanecta runs on
three tables.

Every concept in the system — entities, relationships, types, permissions, comments — is stored in three tables. Everything else is a projection derived from them.

The source of truth

No matter what a user creates or how the schema evolves, the underlying storage never changes shape. Three tables, three responsibilities.

items
What exists

Every entity in the system — a note, a task, a relationship, a permission, a type definition — is an item. The type field distinguishes them; a JSONB payload carries their data.

item_history
What existed

A full snapshot of every item is written here on every change. Not just what changed — the whole row. This gives time-travel queries, audit trails, and disaster recovery in one place.

activity
What happened

An append-only event log of everything that was done: views, edits, permission grants, failed access attempts. Separate from history because behaviour and state are different concerns.

Why this works

Kanecta's type system is open-ended — users define their own types at runtime. You cannot pre-define a schema per type. A single flexible table is not a compromise; it is the only correct answer for a system like this.

"The type universe in Kanecta is genuinely open-ended — users define types. You can't define a schema per type upfront. This is the actual reason a single items table makes sense."

Each of the three tables has a distinct access pattern and lifecycle. Items are queried by id and type. History is queried by item and time range. Activity is queried by actor, verb, and time — closer to a time series. Keeping them separate keeps the indexes clean and retention policies independent.

Everything else is a projection

A projection is a derived read-model: shaped for a specific query, built from the three source tables, and fully disposable. If a projection is lost or corrupted, it is rebuilt from the source. No projection is ever the source of truth.

items where type = 'relationship'
→ Tree index — fast parent/child traversal
items where type = 'permission'
→ Auth cache — sub-millisecond access checks
activity where verb = 'like'
→ Like count — denormalised for display
item payload fields
→ Search index — full-text across all types

This is the CQRS read-model pattern: one write model, multiple read models. Projections are the performance layer. The three tables are the truth layer. The discipline required is that no application code ever writes to a projection as if it were authoritative.

The one rule

Invariant

Never let a projection become a source of truth. The moment code writes to a projection without also writing to items, the model is broken. Every other property of this architecture depends on this boundary holding.

This boundary is enforced by convention, code review, and lint tooling. It is the only invariant the architecture asks you to protect.
