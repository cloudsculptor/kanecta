== Entry points

[cols="1,3"]
|===
| Entry point | Path

| Studio
| `kanecta-app-studio` → `kanecta-api-client` → `kanecta-api` → `kanecta-lib` + `kanecta-ai/kanecta-claude-bridge`

| Hosted functions (inside Studio or kanecta-runner)
| user code → `kanecta-sdk` → `kanecta-api-client` → `kanecta-api` → …

| Runner (pipeline / standalone)
| `kanecta-runner` → hosted functions → `kanecta-sdk` → `kanecta-api-client` → `kanecta-api` → …

| MCP server
| `kanecta-mcp` → `kanecta-lib`

| CLI
| `kanecta-cli` → `kanecta-lib`
|===

== Storage adapters

`kanecta-lib` selects a storage adapter at runtime:

[cols="1,3"]
|===
| Adapter | When to use

| `kanecta-filesystem`
| Default ("easy start"). Items and files both live as local JSON/binary files under `~/.kanecta/`. Human-readable, zero config, zero external dependencies.

| `kanecta-cloud`
| "Bring your own" cloud/remote mode. A single adapter `kanecta-lib` selects in place of `kanecta-filesystem` — internally composes `kanecta-database` (items, via `kanecta-postgres`) and `kanecta-s3` (files), so the rest of the system doesn't need to know storage is split across two backends. Postgres and the S3-compatible store may be self-hosted (incl. local, e.g. native Postgres + MinIO for dev/test) or fully managed cloud services — `kanecta-cloud` doesn't need to know which.
|===

NOTE: Even in `kanecta-cloud` mode, some operations still touch the local filesystem out of necessity — e.g. user-provided functions are pulled from Postgres/S3 and written to a local working directory before being executed in a sandboxed process, then cleaned up. "Cloud mode" means items/files aren't stored on local disk as the source of truth, not that the filesystem is never touched.
