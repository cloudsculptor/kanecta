# Work with function items

---

## Overview

`function` is a primitive item type in Kanecta. Unlike other primitives, a function item stores a rich structured definition in a companion `function.json` file (alongside `metadata.json`) and, on every save via the Studio, generates a runnable TypeScript scaffold in a `function/` subdirectory.

---

## The `function.json` file spec

**Source of truth:** `kanecta-specification/1.2.0/file-specs/function.json`
**Package export:** `import { functionSpec } from '@kanecta/specification'`

Stored at: `.kanecta/data/{s1}/{s2}/{uuid}/function.json`

### Key fields

| Field | Required | Description |
|---|---|---|
| `parameters` | yes | Ordered array — each entry must have `name` and exactly one of `type` or `typeId` |
| `returnType` | conditional | TypeScript primitive return type. Required if `returnTypeId` absent |
| `returnTypeId` | conditional | UUID of a Kanecta type for the return value. Required if `returnType` absent |
| `description` | no | JSDoc summary |
| `async` | no | `true` if the function is async (default `false`) |
| `ai` | no | `true` if the function invokes AI internally (default `false`) |
| `skill` | no | UUID of a Kanecta skill item; function delegates to it instead of (or in addition to) `body` |
| `typeParameters` | no | Generic type params — each has `name`, optional `constraint`, optional `default` |
| `throws` | no | Array of `{ type, description? }` |
| `deprecated` | no | Deprecation notice string |
| `body` | no | Implementation as a source string. Omit for signature-only definitions |

### Parameter fields

| Field | Required | Description |
|---|---|---|
| `name` | yes | Parameter name |
| `type` | conditional | **TypeScript primitive only** — `string`, `number`, `boolean`, `null`, `undefined`, `void`, `never`, `unknown`, `any`, `symbol`, `bigint`, and compositions (`string[]`, `string \| null`, `Promise<T>`). **No object types** — named or inline. Mutually exclusive with `typeId` |
| `typeId` | conditional | UUID of a Kanecta type. For all object types. Resolved to its JSON schema at runtime. Mutually exclusive with `type` |
| `optional` | no | Whether parameter is optional (`?` suffix). Default `false` |
| `rest` | no | Whether it is a rest parameter (`...args`). Default `false` |
| `defaultValue` | no | Default value expression as a source string (e.g. `"200"`, `"[]"`) |
| `description` | no | JSDoc `@param` description |

### The `type` vs `typeId` rule

- **Primitive TypeScript types** (`string`, `number`, `boolean`, primitives and their compositions) → use `type`
- **Any object type** (custom types, named interfaces, `Record<K,V>` etc.) → use `typeId` pointing to a Kanecta type UUID
- Inline object shapes (`{ name: string }`) are **not permitted** in the `type` field

## Files on disk

After saving a function item via the Edit Function dialog:

```
.kanecta/data/{s1}/{s2}/{uuid}/
  metadata.json          ← item metadata (always present)
  function.json          ← function spec (written by Studio save)
  function/
    package.json         ← written once; name = kanecta-fn-{slug}
    tsconfig.json        ← written once; strict ES2020 commonjs
    index.ts             ← always regenerated from function.json
```

`package.json` and `tsconfig.json` are written once and not overwritten on re-save. `index.ts` is always regenerated.

### What `index.ts` contains

```typescript
// Generated by Kanecta — regenerated on each save.

// Interface for each Kanecta typeId param/return (resolved from .kanecta/types/)
interface ContactType {
  name: string;
  email?: string;
}

/**
 * Description from function.json.
 * @param id - param description
 * @throws {NotFoundError} - when thrown description
 */
export async function fetchUser<T extends User = User>(
  id: string,
  options?: ContactType,
): Promise<T> {
  // body injected verbatim, or "// TODO: implement"
}
```

Kanecta `typeId` params are resolved by reading `.kanecta/types/{s1}/{s2}/{typeId}/type.json` and generating a TypeScript `interface` from `jsonSchema.properties`.

## API endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/items/:id/function` | Read `function.json` for the item. 404 if not present |
| `PUT` | `/items/:id/function` | Write `function.json` and regenerate the `function/` scaffold |
| `GET` | `/items/:id/function/scaffold` | Returns `{ exists: boolean }` — whether `function/` directory exists |

## Studio — Edit Function dialog

`kanecta-ui/kanecta-apps/kanecta-app-studio/src/components/views/TreeView/EditFunctionDialog.tsx`

- Opened via the **Edit** (pencil) icon in the function-type action group at the left of TreeNode-actions
- Loads `function.json` via `getFunctionData(id)` and `function/scaffold` exists check on open
- Form mirrors the `function.json` spec; uses `functionSpec` from `@kanecta/specification` for field descriptions as helper text
- Parameter type mode toggle: **Primitive** (free text, no object types) / **Kanecta type** (UUID field)
- Return type mode toggle: same as above

### Save flow (two-stage)

**Stage 1 — dirty warning:** If the form has any unsaved changes (`isDirty`) AND the `function/` scaffold already exists on disk, shows a warning:
> "Saving will regenerate `index.ts`. Manual edits will be overwritten."
→ Cancel / Save anyway

**Stage 2 — body conflict:** Re-fetches current disk `function.json`. If disk body ≠ form body, shows `BodyConflictDialog`:
- Left panel: current body on disk
- Right panel: body in the form
- **Use disk** → restores disk body into the form (no auto-save)
- **Use mine** → saves form body to disk

## Studio — Run Function dialog

`kanecta-ui/kanecta-apps/kanecta-app-studio/src/components/views/TreeView/RunFunctionDialog.tsx`

- Opened via the **Run** (green Fab) in the function-type action group, or via `FunctionLinksButton`
- Loads `function.json`, renders one input field per parameter
- Required params without `defaultValue` must be filled before Run is enabled
- `defaultValue` params are pre-populated and don't block Run
- Execution is stubbed — wire `handleRun` to the execution engine

## Wiring types to functions — `meta.functions`

**Source of truth:** `kanecta-specification/1.2.0/file-specs/type.json` → `meta.functions`

In a type's `type.json`, the `meta` block can include:

```json
"functions": {
  "consumedBy": ["<function-item-uuid>", ...],
  "producedBy": ["<function-item-uuid>", ...]
}
```

| Field | Meaning |
|---|---|
| `consumedBy` | UUIDs of `function` primitive items that accept this type as a parameter |
| `producedBy` | UUIDs of `function` primitive items that return this type |

These are stored in the type's `type.json` under `.kanecta/types/`. Update them when a function's signature changes to add or remove this type.

## FunctionLinksButton

`kanecta-ui/kanecta-apps/kanecta-app-studio/src/components/views/TreeView/FunctionLinksButton.tsx`

Rendered at the left of TreeNode-actions for any item with a `typeId` (`item.type === 'object'`).

- On mount, fetches the type schema via `GET /types/:typeId/schema`
- If `meta.functions.consumedBy` or `meta.functions.producedBy` has entries, resolves each UUID to its item name via `GET /items/:id`
- Renders a green rounded MUI `ButtonGroup` split button:
  - **Left half** (`▶ FunctionName`): runs the currently selected function by opening `RunFunctionDialog`
  - **Right half** (`▼`): opens a `Popper` menu listing all functions under **Consumed by** / **Produced by** group headers
- Returns `null` while loading or if there are no linked functions — no visible element

## Gotchas

- **`type` field rejects object shapes** — using `{ name: string }` or a named type like `User` in the `type` field will fail spec validation. Always use `typeId` for object types.
- **`index.ts` is always regenerated** — any manual edits to `index.ts` will be overwritten on the next Studio save. Use the body field in `function.json` (via the Edit dialog) to persist implementation code.
- **`package.json` / `tsconfig.json` are written once** — editing them in the IDE is safe; they won't be overwritten.
- **`function.json` body vs `index.ts`** — if you edit `index.ts` directly and then open the Edit dialog, the body conflict dialog will surface the difference.
- **`consumedBy` / `producedBy` are not auto-maintained** — when a function's signature changes, manually update the relevant type's `type.json` to add/remove the function UUID.
- **`FunctionLinksButton` only shows for `object` type items** — function items themselves show the Run/Edit group instead.
