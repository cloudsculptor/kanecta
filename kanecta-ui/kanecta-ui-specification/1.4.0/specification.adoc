== Overview

This document defines the rules for every package in `kanecta-ui/`. It is the UI-layer equivalent of `kanecta-core/kanecta-specification/`. Tooling in `kanecta-ui-validator` enforces these rules programmatically.

IMPORTANT: The **reference implementation language is TypeScript** and the **styling language is SCSS**. No JavaScript (`.js`/`.cjs`/`.mjs`) and no plain CSS (`.css`) is written — a repository check (`scripts/check-source-languages.sh`, run in CI and pre-commit) fails on any new `.js`/`.css`. Existing `.js`/`.css` is grandfathered tech debt being migrated to TypeScript/SCSS.

The UI layer is divided into two sub-directories:

* `kanecta-components/` — standalone React component packages, one per component or view
* `kanecta-apps/` — consuming applications (Studio, Claude app, platform variants)

A third directory, `kanecta-ui-specification/`, contains this document and the validator.

=== Soft-coded components — the direction

A Connector application is not a fixed, compiled bundle of screens. Its end state is a *thin bootstrapper plus a package of `component` items* that are resolved from the device component store (see the core spec, "The device component store") and rendered at *runtime* — the same way built-in type items are seeded from a manifest rather than hard-coded. "Connector Studio" is simply one such package of component items (its views: tree, table, quality, and the rest); a different Connector application is a different package of the same kind of items.

The rules in this document are what make that possible: every component is a standalone package with a stable `component` item identity (`kanecta.item.json`), a props-only API with no store or disk access, and a place in an app's `kanecta.manifest.json`. An application therefore does not import its views by name at compile time — it reads its manifest, resolves each component from the device store by `package@version`, and includes it at runtime. (During the transition a host may still bundle components behind the loader; the manifest → store → item → loader contract is identical, so moving to true runtime loading is a change of loader, not of application code. See <<runtime-resolution>>.)

== `package.json` Rules

Every component package `package.json` must conform to the following rules. The `validateComponentPackage` function in `kanecta-ui-validator` enforces all of these programmatically.

=== Required fields

[cols="1,2,2", options="header"]
|===
| Field | Required value | Reason

| `name`
| `@kanecta/component-<slug>`
| Scoped name ensures uniqueness and allows tree-shaking by consumers.

| `private`
| `true`
| Component packages are workspace-only; they are not independently published to npm.

| `main`
| `src/index.ts` or `src/index.tsx`
| No build step. TypeScript sources are consumed directly by the workspace.

| `types`
| `src/index.ts` or `src/index.tsx`
| TypeScript resolves types from the source entry, not a build artifact.
|===

=== `peerDependencies` rules

All runtime dependencies are peer dependencies. The `dependencies` block must be absent.

* Use `>=` version ranges for all peer dependencies — never `^`. This ensures the consuming app's already-installed version is always accepted without forcing a duplicate install.
* `@kanecta/component-*` workspace packages use `*` (any workspace version).

.Minimum required peer dependency version ranges
[cols="1,1", options="header"]
|===
| Package | Minimum range

| `react`
| `>=19`

| `@mui/material`
| `>=9`

| `@mui/icons-material`
| `>=9`

| `@tanstack/react-query`
| `>=5`

| `@kanecta/component-*`
| `*`
|===

=== `devDependencies` rules

Include only `typescript` and `@types/react`. Do not add MUI or TanStack as devDeps — TypeScript resolves their types through the workspace's installed peerDeps without needing separate type packages.

=== Minimal valid `package.json`

[source,json]
----
{
  "name": "@kanecta/component-my-widget",
  "version": "1.4.0",
  "private": true,
  "type": "module",

== TypeScript Rules

=== Named exports only

No `export default`. Every exported symbol must be named so consumers can tree-shake and so re-exports are unambiguous.

[source,typescript]
----
// ✓ correct
export function ConfidenceBadge({ confidence }: ConfidenceBadgeProps) { ... }
export type { ConfidenceLevel };

// ✗ wrong
export default function ConfidenceBadge(...) { ... }
----

=== No store imports

Component packages must not import from Studio stores (`useWorkspaceStore`, `useReviewStore`, `useUiStore`, `useLocationStore`, etc.) or from any `kanecta-app-*` package. All data flows in through props. This makes components testable in isolation and usable outside Studio.

=== Props-only API

Every stateful dependency — fetch functions, callbacks, current workspace ID, active item ID — must be an explicit prop. The component receives data; it does not reach out to get it.

[source,typescript]
----
// ✓ correct — Studio passes fetch callbacks as props
export interface TodoViewProps {
  onFetchItems: () => Promise<TodoItem[]>;
  onCreateItem: (value: string) => Promise<void>;
  queryKey?: string;
}

// ✗ wrong — component reaches into Studio
import { useWorkspaceStore } from '../../stores/workspaceStore';
----

=== No `?raw` or `?url` imports

Vite query suffixes only resolve at the consuming app's build time. Pass the result as a prop instead.

[source,typescript]
----
// ✓ correct — Studio does the ?raw import and passes the string down
export interface TypesViewProps {
  typeSpec?: string;   // caller passes: import typeSpecRaw from '...?raw'
}

// ✗ wrong — won't resolve outside of Vite's build pipeline
import typeSpecRaw from '../../kanecta-specification/type.json?raw';
----

=== No hooks in a loop

If parallel queries are needed over a dynamic list, use `useQueries` from TanStack Query — not `useQuery` inside `.map()`.

[source,typescript]
----
// ✓ correct
const results = useQueries({
  queries: workspaces.map((ws) => ({
    queryKey: ['items', ws.id],
    queryFn: () => onFetchItems(ws.id),
  })),
});

// ✗ wrong — React hooks violation
const results = workspaces.map((ws) => useQuery(...)); // eslint-disable doesn't fix this
----

=== React context within a package

A package may define a React context internally to avoid deep prop-drilling within a complex component tree. The context must not escape the package boundary — it is an internal implementation detail, not part of the public API.

== SCSS Rules

Kanecta styling is **SCSS only** — never plain `.css`, no PostCSS, no CSS Modules. Styles are component-isolated and travel with the component (important for the soft-coded/device-store model, where a component package is self-contained). Global standards arrive through SCSS `@use` at compile time, never through global stylesheets or the cascade.

=== One co-located `.scss` per component

Exactly one `.scss` file per component, sitting beside it and imported directly:

[source,typescript]
----
import './MyComponent.scss';   // MyComponent.tsx + MyComponent.scss, co-located
----

No shared/global component stylesheet, and no massive global `.scss` — a component's styles live with the component and nowhere else.

=== Modified BEM — block is the component file name

The BEM *block* is not a hand-chosen semantic name: it is the **component's file name as a prefix, PascalCase** (matching the `.tsx` filename). Elements use SCSS `&__element` concatenation; modifiers use `&--modifier`.

[source,scss]
----
// KanectaToolbar.scss
.KanectaToolbar {          // block = the file name
  &__thing { ... }         // compiles to .KanectaToolbar__thing
  &--active { ... }        // compiles to .KanectaToolbar--active
}
----

[source,tsx]
----
<div className="KanectaToolbar">
  <div className="KanectaToolbar__thing" />
</div>
----

`&__element` / `&--modifier` concatenation is the required idiom — it compiles to *flat* classes (`.KanectaToolbar__thing`), so there is still no descendant-selector cascade. Do NOT write descendant nesting (`.Block { .child { … } }`) — only `&`-concatenated BEM.

=== Global standards via `@use`, not the cascade

Shared design decisions — colours, spacing, typography, mixins — live in a shared SCSS **design-system** partial (e.g. `styles/_theme.scss` exporting `$color-*`, spacing/typography scales, and mixins). Each component `@use`s it, so the token *values* are compiled INTO that component's isolated stylesheet:

[source,scss]
----
// _theme.scss  (the design system)
$color-surface: #ffffff;
$space-2: 8px;
@mixin card { border: 1px solid $color-border; border-radius: 8px; }

// KanectaToolbar.scss
@use '../../styles/theme' as t;   // pulled in at compile time — no runtime cascade

.KanectaToolbar {
  background: t.$color-surface;
  &__thing { padding: t.$space-2; @include t.card; }
}
----

Rules:

* **Do NOT theme via global `:root { --color-* }` custom properties consumed through the cascade** — that is "global via cascade" and is exactly what this model avoids.
* **Do NOT hardcode values** (`#ffffff`, `8px`) in a component — pull them from the design system so a change propagates everywhere without touching every file.
* There is **no global stylesheet and no unexpected cascading**; every component is self-contained, and consistency comes from the shared SCSS it imports.

[[component-item]]
== `kanecta.item.json` Rules

Every component package must include a `kanecta.item.json` at its root. This file is a Kanecta item conforming to the core `item.json` spec (v1.4.0) with the following constraints. The `validateComponentItem` function in `kanecta-ui-validator` enforces these programmatically.

=== Identity rules

[cols="1,3", options="header"]
|===
| Field | Rule

| `item.type`
| Must be `"component"`.

| `item.id`
| A stable UUID v4. *Never change it after the package is created* — this UUID is the component's permanent identity in the Kanecta item graph. If a workspace caches this component by ID, changing the UUID breaks the link silently.

| `item.parentId`
| Must be `"00000000-0000-0000-0000-000000000000"` (the well-known root UUID). Component packages live at the root of the item graph, not under any other item.

| `item.value`
| A human-readable display name derived from the slug, title-cased with spaces. Example: `kanecta-component-confidence-badge` → `"Confidence Badge"`.
|===

=== Required meta fields

All fields required by the core `item.json` meta section apply. Additionally:

* `meta.specVersion` — must be `"1.4.0"`.
* `meta.files.body` — must be `"src/index.ts"` (or `.tsx`). Points to the sidecar containing the component source entry.

=== Payload

`payload` must be present and conform to the `componentPayload` shape from the core spec:

[cols="1,1,3", options="header"]
|===
| Field | Required | Notes

| `payload.props`
| Yes
| Array of prop definitions (may be empty). Each prop uses the `parameterOrProp` shape — exactly one of `type`, `typeId`, or `functionId`.

| `payload.dependencies`
| No
| Array of npm package names this component imports beyond React. Derived from `peerDependencies` excluding `react`.

| `payload.description`
| No (recommended)
| One-sentence summary of what the component renders.
|===

[[manifest]]
== App Manifest (`kanecta.manifest.json`)

Every app in `kanecta-apps/` that ships or consumes UI components must include a `kanecta.manifest.json` at its root. Its structure mirrors the core `kanecta.manifest.json` pattern. The `validateManifest` function in `kanecta-ui-validator` enforces these rules.

=== Shape

[source,json]
----
{
  "schemaVersion": "1.4.0",
  "package": "kanecta-app-studio",
  "layer": "ui",
  "items": [
    {
      "id": "626c2e16-1608-440a-84c5-cc0a6460ac58",
      "type": "component",
      "file": "../../kanecta-components/kanecta-component-confidence-badge/kanecta.item.json"
    }
  ]
}
----

=== Field rules

[cols="1,3", options="header"]
|===
| Field | Rule

| `schemaVersion`
| Semver string matching the spec version in use (`"1.4.0"`).

| `package`
| The npm package name of this app, without the `@kanecta/` scope.

| `layer`
| Must be `"ui"`. Distinguishes UI manifests from core manifests.

| `items[].id`
| UUID v4. Must match the `item.id` in the referenced `kanecta.item.json`. The manifest never assigns its own IDs — it mirrors the component's canonical identity.

| `items[].type`
| Kanecta item type string (typically `"component"`).

| `items[].file`
| Path to the `kanecta.item.json`, relative to this manifest file. No duplicate IDs allowed.
|===

[#runtime-resolution]
== Runtime Resolution and the Device Store

An application does not hold compile-time references to its views. At startup it *resolves* the components declared in its `kanecta.manifest.json` and includes them dynamically. The resolution contract:

1. *Read the manifest.* Each `items[]` entry names a component by its stable `id` (the `component` item UUID) and points at its `kanecta.item.json`.
2. *Resolve from the device store.* Each component package is materialised in the per-device component store (core spec, "The device component store") at `<componentStore>/<package>@<version>/`. The application resolves the package by name + version; multiple versions may be present and an app binds to the one its manifest depends on.
3. *Load and render.* The host loads the component's entry (`meta.files.body`, a named export) and renders it with props (below). The physical load mechanism is host-specific and MUST NOT be assumed by the component: during the transition a browser host may bundle components behind a loader; a host with a build/serve step may `import()` built ESM from the store at runtime. Either way the component sees the same props-only interface.

Because identity is the `component` item UUID (not an import path), a component can be added, removed, reordered, or swapped for a different version by changing *data* (the manifest / the store) — with no application code change. This is what "the app is a package of component items" means in practice.

*Populating the store.* The store is filled from manifests. Running from source (the Connector repository), the packages under `kanecta-ui/kanecta-components/` are synced into the store keyed `package@version`. Running from a registry (future), packages are fetched by the same key. An application never reads `kanecta-ui/kanecta-components/` directly at runtime — always the store — so the two modes are indistinguishable to it.

== View Components and the Host State Contract

Component packages fall into two broad classes, and one class carries an extra contract.

* *Display components* — badges, chips, indicators, bars. Pure functions of their props; no state of their own.
* *View components* — full-screen views (tree, table, quality, …). Stateful and long-lived, but *the host owns that state, not the view*.

A view MUST NOT persist anything itself. It does not know its environment (filesystem, browser, Electron), so it cannot and must not touch disk, `localStorage`, or any store — consistent with the props-only / no-store-imports rules above. Instead:

* The *host application* owns per-view state and its persistence. Connector Studio, for example, keeps the state of each view a user has opened and restores it on return, so the user can switch between views without losing place. Only one view is mounted at a time — opening a view replaces the current one, it is not layered on top.
* A view receives its state and a way to update it *through props*:
** `state` — the view's last-persisted state (opaque to the host; the view defines its own shape). Absent/`null` on first open.
** `onStateChange(next)` — the view calls this when its state changes; the host persists it (to file, or wherever that host keeps state) and passes it back as `state` next time.
* A view also receives the *Kanecta data interface* it needs (the API / query client) through props — never by importing a store — so the same view works unchanged across datastores and hosts.

This keeps view components portable and host-agnostic: the same `component` item renders in Studio, in another Connector app, or in a future runtime, with each host free to persist state however it likes.

== Storybook

Every component package that exports a renderable React component must include at least one Storybook story file alongside the component source:

* File: `src/<ComponentName>.stories.tsx`
* Must include at least one `Default` story exercising the happy path with realistic props.
* Stories must not import from Studio or any `kanecta-app-*` package.
* Stories use CSF3 format (`@storybook/react`).

Shared display components (badge, chip, indicator, bar) must additionally include a story for each meaningful visual state — for example, a story per `ConfidenceLevel` value for `ConfidenceBadge`.

=== Package versioning and cache identity

Two distinct notions of identity apply to a component, and they must not be conflated:

* *Item identity* — the `component` item's UUID in `kanecta.item.json`. This is *permanent* and stable across every version and every installation; it is how the item graph and any manifest reference the component. It never changes.
* *Package version* — the component package's `package.json` `version`. Together with the package name it forms the *cache key*, `package@version`, under which the package is materialised in the device component store (core spec, "The device component store").

So a component has *one* permanent UUID and *many* possible `package@version` artifacts. The device store may hold several versions of the same package side by side, and different applications on the same device may bind to different versions of the same component — all sharing the one item UUID. Changing a component's behaviour is a *new version* (a new cache artifact); it is never a new UUID. Conversely, the UUID is the thing an app's manifest pins, and the version is what the resolver selects.
