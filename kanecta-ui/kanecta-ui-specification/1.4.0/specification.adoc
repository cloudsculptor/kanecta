== Overview

This document defines the rules for every package in `kanecta-ui/`. It is the UI-layer equivalent of `kanecta-core/kanecta-specification/`. Tooling in `kanecta-ui-validator` enforces these rules programmatically.

IMPORTANT: The **reference implementation language is TypeScript** and the **styling language is SCSS**. No JavaScript (`.js`/`.cjs`/`.mjs`) and no plain CSS (`.css`) is written ŌĆö a repository check (`scripts/check-source-languages.sh`, run in CI and pre-commit) fails on any new `.js`/`.css`. Existing `.js`/`.css` is grandfathered tech debt being migrated to TypeScript/SCSS.

The UI layer is divided into two sub-directories:

* `kanecta-components/` ŌĆö standalone React component packages, one per component or view
* `kanecta-apps/` ŌĆö consuming applications (Studio, Claude app, platform variants)

A third directory, `kanecta-ui-specification/`, contains this document and the validator.

[[package-json-rules]]
== `package.json` Rules

Every component package `package.json` must conform to the following rules. The `validateComponentPackage` function in `kanecta-ui-validator` enforces all of these programmatically.

=== Required fields

[cols="1,2,2", options="header"]
|===
| Field | Required value | Reason

| `name`
| `@kanecta/component-<slug>`
| Scoped name ensures uniqueness and allows tree-shaking by consumers.

| `private`
| `true`
| Component packages are workspace-only; they are not independently published to npm.

| `main`
| `src/index.ts` or `src/index.tsx`
| No build step. TypeScript sources are consumed directly by the workspace.

| `types`
| `src/index.ts` or `src/index.tsx`
| TypeScript resolves types from the source entry, not a build artifact.
|===

=== `peerDependencies` rules

All runtime dependencies are peer dependencies. The `dependencies` block must be absent.

* Use `>=` version ranges for all peer dependencies ŌĆö never `^`. This ensures the consuming app's already-installed version is always accepted without forcing a duplicate install.
* `@kanecta/component-*` workspace packages use `*` (any workspace version).

.Minimum required peer dependency version ranges
[cols="1,1", options="header"]
|===
| Package | Minimum range

| `react`
| `>=19`

| `@mui/material`
| `>=9`

| `@mui/icons-material`
| `>=9`

| `@tanstack/react-query`
| `>=5`

| `@kanecta/component-*`
| `*`
|===

=== `devDependencies` rules

Include only `typescript` and `@types/react`. Do not add MUI or TanStack as devDeps ŌĆö TypeScript resolves their types through the workspace's installed peerDeps without needing separate type packages.

=== Minimal valid `package.json`

[source,json]
----
{
  "name": "@kanecta/component-my-widget",
  "version": "1.4.0",
  "private": true,
  "type": "module",
  "main": "src/index.ts",
  "types": "src/index.ts",
  "scripts": {
    "typecheck": "tsc --noEmit"
  },
  "peerDependencies": {
    "react": ">=19"
  },
  "devDependencies": {
    "@types/react": "^19.0.0",
    "typescript": "~6.0.2"
  }
}
----

== TypeScript Rules

=== Named exports only

No `export default`. Every exported symbol must be named so consumers can tree-shake and so re-exports are unambiguous.

[source,typescript]
----
// Ō£ō correct
export function ConfidenceBadge({ confidence }: ConfidenceBadgeProps) { ... }
export type { ConfidenceLevel };

// Ō£Ś wrong
export default function ConfidenceBadge(...) { ... }
----

=== No store imports

Component packages must not import from Studio stores (`useWorkspaceStore`, `useReviewStore`, `useUiStore`, `useLocationStore`, etc.) or from any `kanecta-app-*` package. All data flows in through props. This makes components testable in isolation and usable outside Studio.

=== Props-only API

Every stateful dependency ŌĆö fetch functions, callbacks, current workspace ID, active item ID ŌĆö must be an explicit prop. The component receives data; it does not reach out to get it.

[source,typescript]
----
// Ō£ō correct ŌĆö Studio passes fetch callbacks as props
export interface TodoViewProps {
  onFetchItems: () => Promise<TodoItem[]>;
  onCreateItem: (value: string) => Promise<void>;
  queryKey?: string;
}

// Ō£Ś wrong ŌĆö component reaches into Studio
import { useWorkspaceStore } from '../../stores/workspaceStore';
----

=== No `?raw` or `?url` imports

Vite query suffixes only resolve at the consuming app's build time. Pass the result as a prop instead.

[source,typescript]
----
// Ō£ō correct ŌĆö Studio does the ?raw import and passes the string down
export interface TypesViewProps {
  typeSpec?: string;   // caller passes: import typeSpecRaw from '...?raw'
}

// Ō£Ś wrong ŌĆö won't resolve outside of Vite's build pipeline
import typeSpecRaw from '../../kanecta-specification/type.json?raw';
----

=== No hooks in a loop

If parallel queries are needed over a dynamic list, use `useQueries` from TanStack Query ŌĆö not `useQuery` inside `.map()`.

[source,typescript]
----
// Ō£ō correct
const results = useQueries({
  queries: workspaces.map((ws) => ({
    queryKey: ['items', ws.id],
    queryFn: () => onFetchItems(ws.id),
  })),
});

// Ō£Ś wrong ŌĆö React hooks violation
const results = workspaces.map((ws) => useQuery(...)); // eslint-disable doesn't fix this
----

=== React context within a package

A package may define a React context internally to avoid deep prop-drilling within a complex component tree. The context must not escape the package boundary ŌĆö it is an internal implementation detail, not part of the public API.

== SCSS Rules

Kanecta styling is **SCSS only** ŌĆö never plain `.css`, no PostCSS, no CSS Modules. Styles are component-isolated and travel with the component (important for the soft-coded/device-store model, where a component package is self-contained). Global standards arrive through SCSS `@use` at compile time, never through global stylesheets or the cascade.

=== One co-located `.scss` per component

Exactly one `.scss` file per component, sitting beside it and imported directly:

[source,typescript]
----
import './MyComponent.scss';   // MyComponent.tsx + MyComponent.scss, co-located
----

No shared/global component stylesheet, and no massive global `.scss` ŌĆö a component's styles live with the component and nowhere else.

=== Modified BEM ŌĆö block is the component file name

The BEM *block* is not a hand-chosen semantic name: it is the **component's file name as a prefix, PascalCase** (matching the `.tsx` filename). Elements use SCSS `&__element` concatenation; modifiers use `&--modifier`.

[source,scss]
----
// KanectaToolbar.scss
.KanectaToolbar {          // block = the file name
  &__thing { ... }         // compiles to .KanectaToolbar__thing
  &--active { ... }        // compiles to .KanectaToolbar--active
}
----

[source,tsx]
----
<div className="KanectaToolbar">
  <div className="KanectaToolbar__thing" />
</div>
----

`&__element` / `&--modifier` concatenation is the required idiom ŌĆö it compiles to *flat* classes (`.KanectaToolbar__thing`), so there is still no descendant-selector cascade. Do NOT write descendant nesting (`.Block { .child { ŌĆ” } }`) ŌĆö only `&`-concatenated BEM.

=== Global standards via `@use`, not the cascade

Shared design decisions ŌĆö colours, spacing, typography, mixins ŌĆö live in a shared SCSS **design-system** partial (e.g. `styles/_theme.scss` exporting `$color-*`, spacing/typography scales, and mixins). Each component `@use`s it, so the token *values* are compiled INTO that component's isolated stylesheet:

[source,scss]
----
// _theme.scss  (the design system)
$color-surface: #ffffff;
$space-2: 8px;
@mixin card { border: 1px solid $color-border; border-radius: 8px; }

// KanectaToolbar.scss
@use '../../styles/theme' as t;   // pulled in at compile time ŌĆö no runtime cascade

.KanectaToolbar {
  background: t.$color-surface;
  &__thing { padding: t.$space-2; @include t.card; }
}
----

Rules:

* **Do NOT theme via global `:root { --color-* }` custom properties consumed through the cascade** ŌĆö that is "global via cascade" and is exactly what this model avoids.
* **Do NOT hardcode values** (`#ffffff`, `8px`) in a component ŌĆö pull them from the design system so a change propagates everywhere without touching every file.
* There is **no global stylesheet and no unexpected cascading**; every component is self-contained, and consistency comes from the shared SCSS it imports.

[[component-item]]
== `kanecta.item.json` Rules

Every component package must include a `kanecta.item.json` at its root. This file is a Kanecta item conforming to the core `item.json` spec (v1.4.0) with the following constraints. The `validateComponentItem` function in `kanecta-ui-validator` enforces these programmatically.

=== Identity rules

[cols="1,3", options="header"]
|===
| Field | Rule

| `item.type`
| Must be `"component"`.

| `item.id`
| A stable UUID v4. *Never change it after the package is created* ŌĆö this UUID is the component's permanent identity in the Kanecta item graph. If a workspace caches this component by ID, changing the UUID breaks the link silently.

| `item.parentId`
| Must be `"00000000-0000-0000-0000-000000000000"` (the well-known root UUID). Component packages live at the root of the item graph, not under any other item.

| `item.value`
| A human-readable display name derived from the slug, title-cased with spaces. Example: `kanecta-component-confidence-badge` ŌåÆ `"Confidence Badge"`.
|===

=== Required meta fields

All fields required by the core `item.json` meta section apply. Additionally:

* `meta.specVersion` ŌĆö must be `"1.4.0"`.
* `meta.files.body` ŌĆö must be `"src/index.ts"` (or `.tsx`). Points to the sidecar containing the component source entry.

=== Payload

`payload` must be present and conform to the `componentPayload` shape from the core spec:

[cols="1,1,3", options="header"]
|===
| Field | Required | Notes

| `payload.props`
| Yes
| Array of prop definitions (may be empty). Each prop uses the `parameterOrProp` shape ŌĆö exactly one of `type`, `typeId`, or `functionId`.

| `payload.dependencies`
| No
| Array of npm package names this component imports beyond React. Derived from `peerDependencies` excluding `react`.

| `payload.description`
| No (recommended)
| One-sentence summary of what the component renders.
|===

[[manifest]]
== App Manifest (`kanecta.manifest.json`)

Every app in `kanecta-apps/` that ships or consumes UI components must include a `kanecta.manifest.json` at its root. Its structure mirrors the core `kanecta.manifest.json` pattern. The `validateManifest` function in `kanecta-ui-validator` enforces these rules.

=== Shape

[source,json]
----
{
  "schemaVersion": "1.4.0",
  "package": "kanecta-app-studio",
  "layer": "ui",
  "items": [
    {
      "id": "626c2e16-1608-440a-84c5-cc0a6460ac58",
      "type": "component",
      "file": "../../kanecta-components/kanecta-component-confidence-badge/kanecta.item.json"
    }
  ]
}
----

=== Field rules

[cols="1,3", options="header"]
|===
| Field | Rule

| `schemaVersion`
| Semver string matching the spec version in use (`"1.4.0"`).

| `package`
| The npm package name of this app, without the `@kanecta/` scope.

| `layer`
| Must be `"ui"`. Distinguishes UI manifests from core manifests.

| `items[].id`
| UUID v4. Must match the `item.id` in the referenced `kanecta.item.json`. The manifest never assigns its own IDs ŌĆö it mirrors the component's canonical identity.

| `items[].type`
| Kanecta item type string (typically `"component"`).

| `items[].file`
| Path to the `kanecta.item.json`, relative to this manifest file. No duplicate IDs allowed.
|===

== Storybook

Every component package that exports a renderable React component must include at least one Storybook story file alongside the component source:

* File: `src/<ComponentName>.stories.tsx`
* Must include at least one `Default` story exercising the happy path with realistic props.
* Stories must not import from Studio or any `kanecta-app-*` package.
* Stories use CSF3 format (`@storybook/react`).

Shared display components (badge, chip, indicator, bar) must additionally include a story for each meaningful visual state ŌĆö for example, a story per `ConfidenceLevel` value for `ConfidenceBadge`.

== Validator Reference

The `kanecta-ui-validator` package is at `kanecta-ui-specification/1.4.0/kanecta-ui-validator/`. Import via the `@kanecta/ui-specification` package:

[source,javascript]
----
import { validateComponentItem, validateManifest, validateComponentPackage }
  from '@kanecta/ui-specification/validator';
----

All three functions return `{ valid: boolean, errors: ValidationError[] }` where each error has:

* `path` ŌĆö dot-notation path to the offending field (e.g. `"item.type"`, `"payload.props[0].name"`)
* `message` ŌĆö human-readable explanation
* `rule` ŌĆö machine-readable rule ID (e.g. `"kanecta-ui:component-type"`, `"format:uuid"`)

[cols="2,3", options="header"]
|===
| Rule ID | Meaning

| `kanecta-ui:component-type`
| `item.type` must be `"component"`

| `kanecta-ui:non-root-id`
| `item.id` must not be the all-zeros root UUID

| `kanecta-ui:root-parent`
| `item.parentId` must be the all-zeros root UUID

| `kanecta-ui:files-required`
| `meta.files` must be present

| `kanecta-ui:body-required`
| `meta.files.body` must be set

| `kanecta-ui:body-path`
| `meta.files.body` must start with `src/`

| `kanecta-ui:payload-required`
| `payload` must be present for component items

| `kanecta-ui:props-required`
| `payload.props` must be an array

| `kanecta-ui:layer`
| Manifest `layer` must be `"ui"`

| `kanecta-ui:duplicate-id`
| Manifest item IDs must be unique

| `kanecta-ui:component-name`
| Package `name` must match `@kanecta/component-<slug>`

| `kanecta-ui:private`
| Package `private` must be `true`

| `kanecta-ui:main-src`
| Package `main` must point into `src/`

| `kanecta-ui:no-dependencies`
| `dependencies` block must be absent

| `kanecta-ui:peer-deps-required`
| `peerDependencies` must be present

| `kanecta-ui:react-peer`
| `react` must be listed as a peer dependency

| `kanecta-ui:peer-range`
| All peer ranges must use `>=` not `^`
|===
